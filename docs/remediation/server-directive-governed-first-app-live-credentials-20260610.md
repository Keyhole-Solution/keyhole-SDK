## Executive Summary

CE-V5-S51-C02 is no longer blocked on credentials. Device login through the CLI
successfully authenticated against the production realm and produced a usable
local credential. `keyhole whoami --json` confirmed a real production identity.

The remaining blocker is server-side MCP surface availability. The live verifier
fails closed during capabilities discovery because production MCP does not
advertise the governed first-app repo registration operation:

```text
MCP capabilities missing required operation(s): repo.register
```

C-02 cannot close until the production MCP boundary can execute:

```text
my-first-app -> live MCP registration -> live context compile -> governed run -> Event Spine/proof receipt
```

## Required Server Actions

1. Add or expose a governed repo registration operation for local first-app participants.
2. Ensure `GET /mcp/v1/capabilities` advertises that operation using a client-discoverable operation ID.
3. Ensure context compile is discoverable as `context.compile` or as a documented equivalent that the SDK can map safely.
4. Ensure the governed runtime path returns real Event Spine/proof evidence when the run is accepted.
5. Confirm the production device-login credential has sufficient authority for this flow, or return an explicit authorization error with required scope guidance.

## Required Capability Contract

The SDK currently recognizes operations from either the top-level
`operations` field or recursively discovered run types. Backend should expose
one of the following shapes.

Preferred explicit operation descriptors:

```json
{
  "operations": [
    {
      "operation_id": "repo.register",
      "name": "repo.register",
      "method": "POST",
      "path": "/mcp/v1/repos/register"
    },
    {
      "operation_id": "context.compile",
      "name": "context.compile",
      "method": "POST",
      "path": "/mcp/v1/runs/start"
    }
  ]
}
```

Acceptable map shape:

```json
{
  "operations": {
    "repo.register": {
      "method": "POST",
      "path": "/mcp/v1/repos/register"
    },
    "context.compile": {
      "method": "POST",
      "path": "/mcp/v1/runs/start"
    }
  }
}
```

If backend chooses different canonical operation IDs or paths, provide the exact
IDs and paths so the SDK can add a non-bypassing compatibility mapper. Do not
require the client to guess from private server internals.

## Required `repo.register` Semantics

The operation must accept the local first-app declaration bundle generated by
the SDK. The request shape is:

```json
{
  "repo": {
    "name": "my-first-app",
    "path_digest": "<sha256 of local path string>"
  },
  "native_artifacts": {
    "keyhole": {},
    "governance_contract": {},
    "capability_passport": {},
    "dependencies": {}
  }
}
```

The artifact objects are populated from these files when present:

```text
my-first-app/keyhole.yaml
my-first-app/governance_contract.yaml
my-first-app/capability_passport.yaml
my-first-app/dependencies.yaml
```

The response must include one stable identifier:

```text
registration_id
repo_id
id
result.registration_id
```

If registration is rejected, return a structured reason that distinguishes:

```text
auth/scope failure
invalid declaration schema
participant already registered
repo identity conflict
unsupported local-only repo
server policy rejection
```

## Required `context.compile` Semantics

The SDK sends:

```json
{
  "run_type": "context.compile",
  "params": {
    "repo": "my-first-app",
    "registration_id": "<repo.register response id>"
  }
}
```

The response must include a stable governed context reference in at least one of:

```text
governance_context_id
ctxpack_digest
digest
ctx_ref_sha256
result.governance_context_id
result.ctxpack_digest
data.governance_context_id
data.ctxpack_digest
keyhole.ctx_ref_sha256
```

If context compile is asynchronous, return enough run/status information for the
SDK to poll without inventing state. If the server requires a different async
contract, provide the canonical polling operation and response fields.

## Required Governed Run / Runtime Receipt Evidence

After local invariant proof passes, the runtime submits a governed realization
with:

```json
{
  "candidate_digest": "sha256:<computed>",
  "require_governed": true,
  "governance_context_id": "<compiled context id>",
  "local_invariant_result": {
    "verdict": "ACCEPT"
  }
}
```

The acceptance receipt must contain real upstream evidence:

```text
governed=true
event_spine_evidence=true
governance_verdict=ACCEPT
drift_state=<clean/non-drifted/current equivalent>
governance_context_id=<real id>
mcp_event_id or event pointer=<real id>
```

Optional evidence fields should be returned when available:

```text
receipt_id=<real id>
proof_id=<real id>
passport_digest=<real digest>
trust_digest=<real digest>
```

If the MCP server accepts governance but cannot yet provide Event Spine/proof
evidence, return an explicit missing-evidence condition. The client must not
claim `event_spine_evidence=true` without a real upstream receipt.

## Capability Advertisement Must Match Implementation

Do not advertise `repo.register` or `context.compile` until the corresponding
operation actually works for the production realm and the device-login identity
class used by the CLI.

Conversely, if an equivalent operation already exists under a different name,
advertise an alias or document the canonical mapping. The SDK can be updated to
use an equivalent public MCP boundary operation, but it cannot bypass the MCP
boundary or rely on undocumented server internals.

## Security and Governance Requirements

- Keep authentication in `kh-prod` unless backend explicitly provides another realm.
- Do not ask clients to paste or commit bearer tokens.
- Device login through `keyhole login --flow device` is the preferred future connection path.
- The verifier may use the persisted device-login credential via token refresh; `KEYHOLE_MCP_TOKEN` is not required when a valid credential exists.
- Do not manually mutate Event Spine, ATP, controller state, or evidence stores to satisfy C-02.
- All evidence IDs must be produced by the live MCP/runtime path.
