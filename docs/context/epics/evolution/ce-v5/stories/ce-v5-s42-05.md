CE-V5-S42-05 — Governed Context Retrieval Bootstrap
Goal

Add the first context-retrieval flow to the developer kit so it can consult governed platform truth before implementation or dispatch.

Why It Matters

This is where the developer kit stops behaving like a static SDK and starts behaving like a governed participant.

A static SDK ships helpers.

A governed participant asks the live system for truth before it acts.

That distinction matters.

Without governed context retrieval, the developer kit will still drift toward familiar but non-canonical behaviors:

reading stale docs

inferring architecture from old examples

assuming unsupported surfaces exist

guessing topology or contracts from memory

reusing outdated beliefs after the platform changes

That is exactly what the recursive governance model is meant to eliminate.

This story introduces the first real bootstrap flow by which the external participant asks the platform:

what is true now

what surfaces exist now

what boundary rules apply now

what context should be consulted before work begins

This is the first moment where the developer kit becomes boundary-aware in behavior, not just in documentation.

Story Thesis

Capabilities discovery teaches the participant what the boundary is.

Governed context retrieval teaches the participant what the governed system currently knows about itself.

The correct external participant sequence therefore becomes:

discover the live boundary through capabilities

bootstrap auth/identity posture where needed

retrieve governed context through the published context-access surfaces

only then begin implementation, dispatch, or deeper interaction

This story creates that third step.

It gives keyhole-developer-kit its first real context bootstrap primitive so it can ask the platform for contextual truth rather than inventing it locally.

This story is about context consumption through the boundary, not private-source retrieval and not runtime mutation.

Problem Statement

The developer kit now has, or is gaining:

an external participant boundary constitution

rehydrated agent instructions

a capabilities discovery client

auth and identity bootstrap guidance

But the participant still lacks a first-class way to retrieve governed context before work.

That leaves a dangerous gap:

the participant may know how to connect, but still not know what to believe.

The boundary already discloses current read-only context-access run types.

But until the developer kit can invoke them, normalize them, and teach their use, context remains conceptual rather than operational.

Without this story:

agents will still make architectural assumptions too early

builders will still rely on docs rather than live context

downstream helper flows will lack a stable context primitive

recursive governance will still look partially manual from the participant side

This story closes that gap.

Story Scope

This story covers:

a context bootstrap helper in the developer kit

support for current context-access run types

context snapshot normalization

usage examples showing context retrieval before work

repo guidance telling agents to retrieve context before assumption-making

boundary-safe local representation of retrieved context for downstream usage

This story does not yet cover:

run dispatch helpers

proof submission

contract registration

mutation-bearing participant flows

full recursive demo orchestration

generalized schema discovery for all run types

Those follow later.

Non-Goals

This story must not:

redesign platform context surfaces

expose private platform internals outside the governed boundary

treat local cached context as more authoritative than live retrieval

invent missing context surfaces

hardcode platform source assumptions into the developer kit

implement broad orchestration beyond read-only context retrieval

bypass auth/identity requirements where applicable

This is a context bootstrap story, not a universal client story.

Current Boundary Truth This Story Must Consume

The guidance and implementation in this story must reflect the currently disclosed context-access contract.

A) Context Is Retrieved Through MCP

Context must be retrieved through the boundary, not through private source inspection.

The developer kit must treat MCP as the context entry path.

B) Current Implemented Read-Only Context Surfaces

The current capabilities contract indicates implemented read-only context-access surfaces including:

context.compile

gaps.list

lineage.get.v0_1

convergence.status.v0_1

This story should support the currently implemented set rather than speculating about undeclared future surfaces.

C) Context Is a Governed Truth Aid

Context is not arbitrary metadata.

It is a governed truth surface for external participants and agents before they act.

D) Live Retrieval Remains Authoritative

A retrieved context snapshot can be normalized locally for use, but live boundary retrieval remains authoritative.

E) No Private-Source Dependency

External participants must not need to browse platform source trees to understand platform topology, contracts, or current participant-relevant posture.

Deliverables
1) Context Bootstrap Helper

Implement a first context-retrieval helper in the developer kit.

This helper must:

invoke current read-only context-access surfaces through MCP

operate as a participant-facing read-only bootstrap utility

give downstream consumers a clean, local representation of retrieved context

support context-first workflows before implementation or dispatch

2) Support for Current Context-Access Run Types

Add support for the current read-only context-access run types disclosed by the boundary contract.

At minimum, this includes support for:

context.compile

gaps.list

lineage.get.v0_1

convergence.status.v0_1

The helper should be structured so later additions remain possible without reworking the entire posture.

3) Context Snapshot Normalization

Implement normalization of retrieved context into a stable local representation.

The normalized snapshot should preserve participant-relevant fields such as:

topology

contracts

interfaces

supported context surfaces

relevant guidance

metadata useful for traceability

The goal is not to flatten away truth, but to make downstream usage stable and intelligible.

4) Usage Examples for Retrieving Platform Context Before Work

Add usage examples showing the correct pattern:

discover

bootstrap identity where needed

retrieve governed context

only then begin work, design, or dispatch

These examples must be understandable by both humans and agents.

5) Agent Guidance Reinforcement

Ensure the repo guidance explicitly tells agents to retrieve context before making architectural assumptions, implementation assumptions, or dispatch assumptions.

This story should make the instruction-layer rule executable.

Functional Requirements
FR-1 — Read-Only Context Invocation

The developer kit must be able to invoke the currently implemented read-only context-access surfaces through MCP.

FR-2 — Boundary-Only Context Retrieval

Context retrieval must occur through the boundary and must not depend on reading private platform internals.

FR-3 — Stable Local Representation

Retrieved context responses must be normalizable into a stable local representation usable by downstream helpers and consumers.

FR-4 — Current Surface Support

The helper must support the currently disclosed context-access run types rather than guessing additional undeclared surfaces.

FR-5 — Usage Before Assumption

Repo docs and examples must make it explicit that context retrieval should happen before assumption-making.

FR-6 — Metadata Preservation

Where useful metadata is present in context responses, the normalized local representation should preserve it rather than discarding it unnecessarily.

FR-7 — Conservative Failure Behavior

If context retrieval fails, the helper must fail clearly and conservatively rather than fabricating context.

FR-8 — Minimal Participant-Facing Complexity

The first version should remain small and bootstrap-oriented rather than trying to become a full context orchestration engine.

Implementation Guidance
A) Build on the Capabilities Client

The context bootstrap helper should not ignore the work already established in S42-03.

It should build on the known boundary posture rather than duplicating discovery logic.

Capabilities teaches what context surfaces exist.

This story consumes that knowledge.

B) Support What Exists Now

Implement support for the currently published read-only context surfaces.

Do not guess future context surfaces.

Do not invent naming variants.

C) Normalize for Participant Use

The normalized context representation should focus on what an external participant needs before work:

what system am I talking to

what contracts apply

what interfaces are live

what participant guidance is relevant

what contextual truth should shape next steps

D) Preserve Useful Metadata

Preserve enough retrieval metadata to support:

debugging

snapshot inspection

traceability

later comparison

Do not throw away data that later stories will likely need.

E) Fail Closed

If retrieval fails or response shape is incomplete, the helper should report that clearly.

Do not substitute guessed topology, guessed interfaces, or guessed contracts.

F) Make the “Retrieve Context First” Rule Concrete

This story should make the instruction-layer doctrine real:

before work, retrieve context.

That should show up in:

docs

examples

helper naming

intended usage flow

Supported Initial Context Surface Behavior

The helper should support the current context-access surfaces in a boundary-safe way.

context.compile

Use when the participant needs the broadest current platform context bundle.

This is the primary context bootstrap surface.

gaps.list

Use when the participant needs to browse/search current gaps or understand open work posture.

lineage.get.v0_1

Use when the participant needs causal lineage information for a known target.

convergence.status.v0_1

Use when the participant needs current convergence status posture.

This story does not need to overbuild a large abstraction around them.

It needs to make them callable, normalizable, and usable in a participant-safe way.

Usage Model This Story Should Teach

The developer kit should teach a sequence equivalent to:

Step 1 — Discover

Retrieve capabilities.

Learn:

contract

transport

auth posture

currently implemented context surfaces

Step 2 — Bootstrap Identity Where Needed

Authenticate and inspect identity if the context flow requires authenticated posture.

Step 3 — Retrieve Governed Context

Invoke the appropriate read-only context surface.

Normalize the result locally.

Step 4 — Use Context Before Acting

Only after context retrieval should the participant:

plan implementation

make architectural assumptions

choose run types

prepare deeper governed interaction

That sequence is the beginning of actual recursive governance behavior.

Risks
Risk 1 — Treating Context as Optional

If context retrieval is documented as merely helpful instead of normative, agents will continue to improvise.

This story must make context-first behavior explicit.

Risk 2 — Overfitting to One Surface

If the helper is built only around one context response shape, later surfaces may require unnecessary rework.

Keep the abstraction small but flexible.

Risk 3 — Silent Raw-JSON Leakage

If downstream consumers still need to parse raw response shapes directly, the normalization layer is not doing enough.

Risk 4 — Replacing Live Truth With Cached Beliefs

If local normalized context is treated as more authoritative than live retrieval, truth drift returns.

The design must preserve live-boundary authority.

Risk 5 — Guessing Undeclared Surfaces

The helper must consume current published context surfaces only.

No convention-based guessing.

retrieve live platform context through the boundary, normalize it locally, and use that governed truth before implementation, dispatch, or assumption-making begins.
