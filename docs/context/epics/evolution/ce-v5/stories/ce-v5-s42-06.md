CE-V5-S42-06 — Run-Type Safety & Schema Discovery Helpers
Goal

Prevent humans and agents from guessing workflows, request shapes, or invalid dispatch names.

Why It Matters

The current boundary explicitly requires exact canonical run types. Guessing must become structurally difficult.

The first external participant cannot be launch-grade if it still behaves like this:

“close enough” run-type names are tried optimistically

pluralization is treated as harmless

request bodies are assembled from intuition

dispatch names are inferred from domain nouns

raw boundary errors are the first line of defense

That posture produces unnecessary churn and undermines the platform’s strongest architectural claim:

the boundary is governed, exact, and discoverable.

If the developer kit allows casual guessing, then the participant side teaches the opposite lesson:

discovery is optional

schema is secondary

convention can replace contract

invalid names are normal exploration

That is not acceptable.

This story makes the correct posture operational:

exact canonical run types

schema-aware request construction

dispatch preflight before submission

clear guidance when a name or request shape is invalid

helper behavior driven by capabilities and published client guidance

This is where external participant behavior becomes structurally disciplined rather than merely advised.

Story Thesis

Capabilities discovery tells the participant what the boundary claims.

Context retrieval tells the participant what the governed system currently knows.

Run-type safety and schema discovery complete the next step:

they tell the participant how to ask correctly.

That means the developer kit must stop treating dispatch as an open string field plus arbitrary parameters.

Instead, it must help humans and agents do the following:

discover the current boundary

determine whether a requested run type is canonical

retrieve schema or published guidance where applicable

preflight the dispatch request before sending it

fail early and clearly when the request is malformed, guessed, or unsupported

This story is the discipline layer between “I know the platform exists” and “I am ready to dispatch correctly.”

Problem Statement

The live boundary already teaches an important law:

run types are exact canonical keys

runs are named workflows, not REST resource guesses

guessing will fail

unknown run types should be handled by discovery, not improvisation

The capabilities contract already includes public-safe client guidance such as:

exact run-type discipline

examples of correct and incorrect forms

gap workflow hints

event query selector hints

schema discovery hints where applicable

But if the developer kit does not turn that guidance into actual helper behavior, then users and agents will still:

guess run names

drift into pluralization

invent dispatch keys

send malformed request shapes

learn by trial-and-error against live governance surfaces

That is the wrong ergonomics.

The developer kit must make correctness easier than guessing.

This story creates that safety layer.

Story Scope

This story covers:

run-type validation helpers

schema discovery helper usage

dispatch preflight checks

error-path guidance for unknown run types and invalid request shapes

developer-kit behavior that reflects current capabilities guidance

documentation and examples showing exact run-type discipline

early warnings or refusal behavior before invalid dispatch reaches the boundary where possible

This story does not yet cover:

full proof-bearing flows

contract registration

promotion submission

deep orchestration of all run types

server-side validation redesign

mutation authorization logic

This is a participant-side safety and ergonomics story.

Non-Goals

This story must not:

redesign run dispatch semantics on the server

invent unofficial aliases for canonical run types

silently auto-correct guessed names in ways that hide mistakes

fabricate schema when schema discovery is unavailable

hardcode private implementation details into the developer kit

blur the distinction between published guidance and guaranteed support

bypass boundary validation

This story should make dispatch safer, not more magical.

Current Boundary Truth This Story Must Reflect

The helper behavior and docs in this story must encode the current published client posture.

A) Run Types Are Exact

Run types are exact canonical keys.

They are not:

fuzzy labels

pluralizable guesses

resource names inferred from domain terms

safely discoverable by naming instinct

B) Current Published Guidance Includes Mistake Classes

The current boundary guidance already teaches examples like:

gaps.status ✅

gaps.states ❌

gaps.list ✅

gaps.next_open_canonical ✅

gaps.next ❌

gap.status ❌

The developer kit must reflect this guidance rather than requiring each user to learn it by rejection.

C) Schema Discovery Exists as a Boundary-Safe Concept

Where schema discovery is available, the participant should use it rather than guessing parameter structure.

The current public-safe audit/discovery guidance already indicates schema-oriented discovery as a valid boundary concept.

D) Unknown Run Types Must Fail Conservatively

If a run type is unknown or unsupported, the developer kit must not invent support.

E) Client Guidance from Capabilities Is Real Input

The run-type helpers should reflect published client guidance retrieved from capabilities, not only static repo prose.

Deliverables
1) Run-Type Validation Helpers

Implement helper logic in the developer kit that validates intended run types before dispatch.

This helper must:

accept intended run-type input

compare it against known/published canonical names where available

reject or warn on obvious guessed/invalid names

provide guidance toward the correct canonical name when possible

avoid silently inventing aliases

This is the first participant-side defense against naming drift.

2) Schema Discovery Helper Usage

Implement or document helper usage that retrieves schema or request-shape guidance where applicable before dispatch.

This helper posture must teach:

do not guess parameters

consult schema/guidance first

prefer discovery over convention

The goal is not to fully model every future surface, but to make schema-aware usage the norm.

3) Dispatch Preflight Checks

Add preflight checks before dispatch that verify, where possible:

run type is canonical or plausibly supported

required request shape guidance has been consulted

obviously invalid parameter shapes are caught early

the request is not based on a guessed or malformed dispatch name

This is participant-side preventive discipline before server rejection.

4) Error-Path Guidance for Unknown Run Types and Invalid Request Shapes

Provide clear guidance and helper behavior for error paths such as:

unknown run type

guessed pluralized run type

invalid request shape

missing required parameters

unsupported workflow assumptions

The output should help the user or agent recover by:

re-checking capabilities

consulting schema

using the exact canonical name

avoiding repeated bad dispatch loops

Functional Requirements
FR-1 — Run-Type Validation Before Dispatch

The developer kit must validate intended run types before dispatch where enough information exists to do so safely.

FR-2 — Exact Canonical Guidance

The helpers must guide users toward exact canonical run types and explicitly resist guessed or approximate names.

FR-3 — Schema-Aware Posture

Schema discovery usage must be documented and used where applicable so request shapes are not guessed from convention.

FR-4 — Preflight Safety

Dispatch preflight checks must detect obvious invalid run names or malformed request assumptions before sending requests when possible.

FR-5 — Capability-Guidance Reflection

Client guidance learned from the capabilities contract must be reflected in helper behavior and documentation.

FR-6 — Conservative Failure

When uncertainty remains, helpers must fail clearly or warn clearly rather than fabricating support.

FR-7 — Human and Agent Usability

The safety layer must be understandable and usable by both human contributors and agentic tooling.

FR-8 — No Private-Source Dependency

The helper behavior must not rely on reading private platform source to know what a valid run type is.

Required Output Shape

The exact implementation language may vary, but the participant-side safety layer should conceptually expose these capabilities.

A) Run-Type Check

Given an intended run type, determine:

valid

invalid

unknown

close-to-known with suggestions where safe

B) Suggestion Surface

Where possible, provide safe guidance such as:

“did you mean gaps.status?”

“use gaps.list for browsing, not gaps.get without a specific identifier”

“pluralized form is non-canonical”

Suggestions must be helpful without inventing unsupported surfaces.

C) Schema Hint Surface

Given a run type or intended action, show or retrieve:

known schema shape

known required parameters

known request examples

“schema unavailable, re-discover first” posture where necessary

D) Preflight Result

A preflight result should be able to communicate, at minimum:

pass

warn

reject

reason

suggested next step

E) Error Recovery Guidance

The helper should tell the user or agent to do one or more of:

re-check capabilities

retrieve schema

use exact run-type spelling

stop guessing

switch to the correct canonical workflow

Implementation Guidance
A) Build on Capabilities and Context, Do Not Ignore Them

This story should consume the earlier S42 work:

S42-03 capabilities discovery client

S42-05 governed context retrieval bootstrap

Do not create a separate isolated truth model.

The helpers should rely on boundary-published guidance and current discovery posture.

B) Refuse Guessing by Default

The correct posture is not:

“try it and see”

The correct posture is:

“check first, then dispatch”

The developer kit should make guessing inconvenient and correctness convenient.

C) Suggest Safely, Not Magically

Suggestions are useful, but they must not cross into invention.

A suggestion engine should only recommend names that are grounded in known/published guidance.

D) Treat Schema Discovery as a First-Class Discipline

Even when exact automated schema retrieval is not yet universal across every surface, the docs and helper flow should teach that request shapes are discovered, not improvised.

E) Fail Early Where Possible

Participant-side early warnings save time and reinforce correct behavior.

The goal is not to duplicate all server validation, but to prevent the most common and most wasteful invalid dispatch patterns.

F) Preserve Honest Uncertainty

If the helper does not know, it should say so.

It must not confidently pretend unsupported certainty.

Supported Error Classes This Story Should Help With

The helper behavior and docs should explicitly address at least these categories:

Unknown Run Type

Example:

user or agent provides a run type not present in known guidance

Expected posture:

reject or warn

do not dispatch blindly

Guessed Pluralization / Naming Drift

Example:

gaps.next

Expected posture:

warn or reject

offer grounded alternatives where safe

Wrong Workflow for Intent

Example:

trying to use a detail surface like a listing surface

Expected posture:

explain intended canonical workflow

guide toward correct surface choice

Invalid Request Shape

Example:

required field missing

wrong parameter type or structure

unsupported top-level keys

Expected posture:

point to schema or request example

do not encourage guessing by repeated retries

Usage Model This Story Should Teach

The developer kit should teach a sequence equivalent to:

Step 1 — Discover

Retrieve capabilities.

Learn:

what run-oriented guidance is published

what exact run types are known

what schema discovery posture exists

Step 2 — Retrieve Context if Needed

If architectural context matters, retrieve governed context before deciding the workflow.

Step 3 — Preflight the Dispatch

Before dispatch:

validate run type

consult schema or request example

reject or warn on guessed names

confirm the request shape

Step 4 — Dispatch Only After Safety Check

Only after a valid preflight posture should the request proceed.

This is how participant behavior becomes disciplined rather than improvisational.

Risks
Risk 1 — Over-Correction Into Hidden Magic

If the helper silently rewrites invalid names into valid ones, users may never learn exact boundary discipline.

This story should prefer visible correction guidance over hidden autocorrection.

Risk 2 — Rigid Overfitting to Current Surface Shape

The helper should be precise without being so brittle that every future surface addition forces redesign.

Risk 3 — Duplicating Server Validation Poorly

The goal is not to recreate full server-side enforcement. The goal is to prevent common participant-side mistakes early.

Risk 4 — Treating Partial Guidance as Total Knowledge

If the helper has only partial knowledge, it must say so.

It must not imply exhaustive certainty it does not have.

Risk 5 — Leaving Docs Behind the Code

If the code enforces one posture but the docs still teach ad hoc dispatch, drift will reappear quickly.

Story Completion Statement

When CE-V5-S42-06 is complete, keyhole-developer-kit will no longer behave like a thin wrapper that lets humans and agents throw approximate names and guessed payloads at the boundary until something works.

Instead, it will enforce the correct external participant posture:

discover first, validate the exact workflow name, consult schema where needed, preflight the request, and only then dispatch against the governed MCP boundary.
