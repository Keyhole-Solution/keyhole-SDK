CE-V5-S42-04 — Auth & Identity Bootstrap Guidance
Goal

Provide the correct developer-kit posture for authentication and initial identity inspection.

Why It Matters

External participant onboarding fails immediately if auth and identity are wrong.

A governed external participant cannot safely move from discovery into governed interaction unless it knows:

how authentication actually works

which surfaces are public discovery surfaces

which surfaces require authenticated identity

what whoami is for

when charter and workspace posture begin to matter

the difference between read-only discovery and governed execution

Without that clarity, external usage collapses into avoidable failure modes:

builders try to authenticate against the wrong realm

agents assume discovery and governed usage are the same thing

unauthenticated callers attempt authenticated flows

write-capable intent is attempted before identity posture is confirmed

developers mistake “I can reach the endpoint” for “I am correctly enrolled as a governed participant”

This story fixes that by making the developer kit’s auth and identity bootstrap posture explicit, current, and boundary-aligned.

Story Thesis

The first external participant must not jump directly from “repo exists” to “run things.”

It must bootstrap in the correct order:

discover the live boundary

learn the auth posture

acquire token(s) through the published flow

inspect identity through GET /mcp/v1/whoami

distinguish read-only public discovery from authenticated governed interaction

only then proceed toward context retrieval, run dispatch, and later proof-bearing flows

This story does not yet implement every future participant action.

It teaches the correct entrance posture into the governed system.

That is the first trustworthy external onboarding step.

Problem Statement

The developer kit now has:

a separate repo boundary

agent-instruction rehydration

a capabilities discovery client

But that is not enough.

A participant still needs clear guidance for moving from public discovery into authenticated governed usage.

The live capabilities contract already discloses key facts:

transport is REST/HTTP

auth flow is OIDC/PKCE

realm is keyhole-mcp

GET /mcp/v1/capabilities is a public-safe discovery surface

GET /mcp/v1/whoami is the identity inspection surface

Without a dedicated bootstrap story, those facts remain scattered rather than operationally taught.

The developer kit must teach humans and agents:

when auth is required

how to think about tokens and realm posture

what whoami confirms

how authenticated and unauthenticated flows differ

that read-only discovery is not the same thing as governed action

This story creates that boundary-safe onboarding posture.

Story Scope

This story covers:

documented OIDC/PKCE bootstrap guidance

token acquisition guidance aligned with keyhole-mcp

identity inspection usage guidance for GET /mcp/v1/whoami

clear distinction between unauthenticated discovery surfaces and authenticated governed surfaces

explicit read vs write posture guidance

machine-readable and human-readable bootstrap flow documentation

external participant guidance that prepares later stories without hardcoding unstable internals

This story does not yet cover:

run dispatch implementation

contract registration

proof submission

full workspace provisioning helpers

full charter lifecycle tooling

recursive demo execution

Those come later.

Non-Goals

This story must not:

redesign the platform auth model

expose secrets or credentials in docs

implement hidden or privileged auth shortcuts

bypass charter or workspace rules

imply that authentication alone grants mutation authority

hardcode private enforcement topology into the developer kit

conflate discovery posture with governed execution posture

This is a bootstrap guidance story, not an authority-expansion story.

Current Boundary Truth This Story Must Reflect

The guidance in this story must match the current public-safe capabilities contract and current boundary posture.

A) Public Discovery Surface

The initial discovery surface is:

GET /mcp/v1/capabilities

This surface is:

public-safe

read-only

available before authentication

intended to disclose the boundary contract and connection posture

B) Auth Posture

The live contract indicates:

auth flow: OIDC/PKCE

auth realm: keyhole-mcp

transport: rest-http

The developer kit must treat that as the canonical bootstrap posture.

C) Identity Inspection Surface

The identity inspection surface is:

GET /mcp/v1/whoami

This surface exists to answer the first governed question:

who am I at the boundary right now?

It is not merely a convenience endpoint.

It is the first authenticated truth check.

D) Authenticated vs Unauthenticated Surface Distinction

The guidance must explicitly distinguish:

unauthenticated public discovery

authenticated identity inspection

read-only exploration

governed execution surfaces

later proof-bearing and write-bearing flows

E) Charter / Workspace Posture

The current capabilities contract indicates:

charter required: true

workspace supported: true

This story must not over-implement those flows, but it must prepare the user to understand that authenticated identity is not the whole story for later governed execution.

Deliverables
1) Documented OIDC/PKCE Bootstrap Guidance

Provide explicit developer-kit guidance describing the correct authentication posture for external participants.

This must include:

that the auth model is OIDC/PKCE

that the realm is keyhole-mcp

that public discovery occurs before authentication

that authenticated interaction begins after correct token acquisition

that the guidance is boundary-first, not source-intimacy-driven

2) Token Acquisition Guidance Aligned with keyhole-mcp

Provide token acquisition guidance aligned with the current realm and discovery posture.

This guidance must be usable by:

humans onboarding manually

later helper scripts and bootstrap commands

It must describe the expected relationship between:

discovery

auth

token possession

subsequent identity inspection

3) Identity Inspection Usage Guidance for GET /mcp/v1/whoami

Document how and why a participant uses GET /mcp/v1/whoami.

The guidance must explain that whoami is used to confirm:

authenticated identity posture

subject visibility at the boundary

lane/purpose context where exposed

ownership / participant context where exposed

whether the current bootstrap state is suitable for proceeding

4) Authenticated vs Unauthenticated Surface Distinction

Create a clear guidance section distinguishing:

public discovery surfaces

authenticated identity surfaces

authenticated read surfaces

later governed write or proof-bearing surfaces

This distinction must be explicit enough that a new builder cannot confuse:

“I can call capabilities”
with

“I am fully ready to act as a governed participant”

5) Machine-Readable and Human-Readable Bootstrap Flow

Provide a bootstrap flow that can be consumed both as prose and as a deterministic sequence.

That sequence should conceptually read like:

call capabilities

learn current auth posture

acquire token using the published auth model

call whoami

confirm identity posture

proceed to governed context retrieval or other allowed next steps

Required Guidance Content

The resulting documentation must cover the following concepts clearly.

A) What Can Be Done Before Authentication

The docs must explicitly say that a participant can begin with public discovery:

GET /mcp/v1/capabilities

This is the first truth surface.

B) What Authentication Enables

The docs must explain that authentication is required before:

identity inspection through whoami

later governed interaction

later participant-specific execution flows

C) What whoami Is For

The docs must explain that GET /mcp/v1/whoami is used to confirm current identity posture rather than to discover the whole system.

It is the first authenticated introspection step.

D) Read / Write Distinction

The docs must make clear:

discovery is read-only

identity inspection is read-oriented and boundary-safe

later runs, submissions, and other governed interactions are separate

possession of a token is not the same thing as permission for arbitrary mutation

E) Charter / Workspace Reminder

The docs should state, in boundary-safe language, that later governed flows may require charter and workspace posture, and that authentication alone is not the final participant state.

F) Avoided Anti-Patterns

The docs must warn against:

guessing auth posture from stale docs

assuming every endpoint is publicly callable

confusing public discovery with authenticated execution

skipping whoami and proceeding blindly

treating later write-bearing surfaces as equivalent to read-only exploration

Suggested Bootstrap Flow

The guidance should convey a sequence equivalent to:

Step 1 — Discover

Call:

GET /mcp/v1/capabilities

Learn:

contract

transport

auth flow

auth realm

participant-relevant surfaces

Step 2 — Authenticate

Acquire token(s) according to the published OIDC/PKCE posture for:

realm: keyhole-mcp

Step 3 — Inspect Identity

Call:

GET /mcp/v1/whoami

Confirm:

authenticated boundary identity

visible subject posture

whether the participant is operating in the expected authenticated state

Step 4 — Proceed Carefully

Only after correct identity inspection should the participant move to later flows like:

governed context retrieval

run-type/schema-aware execution

later contract/proof participation

This flow must be described in a way that both humans and agents can follow deterministically.

Implementation Guidance
A) Keep the Guidance Boundary-First

The docs should not instruct users to browse private platform source to understand auth posture.

The boundary already publishes enough truth for correct bootstrap.

B) Avoid Secret-Sharing in Docs

The story should describe the auth posture and token flow shape, not dump sensitive values or local secrets into repo docs.

C) Make whoami Central

A common failure mode is authenticating and then proceeding without identity inspection.

The docs should make whoami the standard first authenticated check.

D) Teach That Discovery Comes First

The external participant should never begin by guessing token or endpoint posture.

It begins with capabilities discovery.

E) Distinguish Bootstrap From Full Governance

The docs should avoid implying that successful auth bootstrap completes the entire participant lifecycle.

It is the first governed entrance step.

Risks
Risk 1 — Overstating Authority

If the docs imply that authentication alone grants broad authority, later participant behavior will drift immediately.

Risk 2 — Underexplaining Identity Inspection

If whoami is treated as an optional extra, builders and agents may proceed with incorrect or partial identity assumptions.

Risk 3 — Blurring Discovery and Governed Usage

If public and authenticated surfaces are not clearly separated, onboarding becomes fragile and error-prone.

Risk 4 — Copying Sensitive Local Auth Lore

The docs must stay boundary-safe and reusable, not encode local secrets or one-off machine assumptions.
