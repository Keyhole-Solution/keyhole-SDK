CE-V5-S42-02 - Copilot / Agent Instruction Rehydration
Goal

Rewrite the repo's formative and legacy agent instructions so they reflect the live MCP boundary rather than stale assumptions.

Why It Matters

This is the first place where agent behavior becomes boundary-governed instead of repo-folklore-driven.

If the developer-kit instructions remain legacy-shaped, then even a well-designed repo split will still fail in practice because agents will continue to behave as if:

platform internals are the first truth surface

run types can be guessed by naming convention

transport/auth posture is inferred from old docs

nested-source intimacy is normal

external participants can learn the platform from stale local assumptions

That is the wrong posture.

The developer kit must teach agents to behave like real governed participants:

discover first

consult capabilities first

retrieve context before assumptions

use exact canonical run types

respect auth, charter, and workspace posture

treat the MCP boundary as the authoritative participant interface

This story is where that posture becomes operational.

Story Thesis

The first external participant repo must not teach agents to "figure it out from the codebase."

It must teach them to:

learn from the live boundary

obey the published contract

avoid guessing

prefer discovery over folklore

prefer deterministic schema/context retrieval over assumption

The canonical agent starting point inside keyhole-developer-kit must therefore become:

GET /mcp/v1/capabilities

not:

private platform source inspection

old comments

stale bootstrap notes

guessed endpoints

guessed run types

This story rehydrates the repo's instruction layer so that future builders and agents start from the same governed truth surface the platform intends external participants to use.

Constitutional Alignment

This story extends and operationalizes existing doctrine. It does not change platform law.

It must remain aligned with:

Rule 00 - Builders Out, Declarations In

MCP Server Boundary

Capability Preservation

Boundary Replay Determinism

This story is about agent instruction posture, not about new runtime powers.

Problem Statement

The repo's current Copilot / agent instructions are formative and legacy.

That means they likely carry one or more of these failure modes:

they were written before the current MCP boundary stabilized

they do not teach capabilities-first discovery

they do not teach exact run-type usage

they may imply source intimacy with the platform

they may omit the current transport/auth posture

they may not distinguish public discovery from authenticated governed interaction

they may fail to explain what must be learned from capabilities vs context vs schema discovery

If left unchanged, the first external participant repo will still generate old behaviors even after the architecture has evolved.

That would make the repo split cosmetic rather than real.

This story fixes that by rewriting the instruction layer so the repo teaches the live boundary correctly.

Story Scope

This story covers:

rewriting the developer-kit agent instructions

establishing an MCP-first instruction set

instructing agents to consult capabilities/context before architectural assumptions

explicitly warning against guessing run types

documenting current transport/auth posture in agent-friendly terms

forbidding stale-source or private-source dependency for platform truth

making the boundary-safe usage model explicit for humans and agents

This story does not yet implement:

the capabilities client

auth bootstrap code

context retrieval helper code

run-type validation helper code

proof submission code

recursive demo orchestration

Those come in later S42 stories.

Non-Goals

This story must not:

redesign the MCP boundary

expose protected internal enforcement topology

instruct agents to infer hidden surfaces from naming conventions

define new run types

give agents permission to bypass discovery/context retrieval

teach agents to depend on platform source code as canonical external truth

hardcode unstable platform internals into the developer-kit instruction layer

This story consumes boundary truth.
It does not redefine it.

Current Boundary Truth This Story Must Encode

The rewritten instructions must reflect the current live posture disclosed through the capabilities contract.

A) Initial Source of Truth

The initial source of truth for external participants is:

GET /mcp/v1/capabilities

This is public-safe, read-only, and the first discovery surface.

B) Current Boundary Contract

The capabilities contract currently indicates:

contract: mcp/v1

transport posture: rest-http

auth flow: OIDC/PKCE

auth realm: keyhole-mcp

minimum SDK version: 0.1.0

charter required: true

workspace supported: true

C) Current Context-Access Contract

Current implemented read-only context-access run types include:

context.compile

gaps.list

lineage.get.v0_1

convergence.status.v0_1

D) Critical Run-Type Rule

Run types are exact canonical keys.

They are:

singular when declared singular

not discoverable by convention

not safely guessable by pluralization or naming instinct

Examples from current guidance include:

gaps.status OK

gaps.states NO

gaps.list OK

gap.status NO

E) Boundary-Safe Discovery Model

Agents must distinguish:

public discovery surfaces

authenticated identity surfaces

run dispatch surfaces

schema/context discovery surfaces

F) Legacy Transport Rule

Legacy SSE and JSON-RPC are tombstoned and must not be used.

reflect live MCP boundary truth

The instructions must explicitly teach:

3) Explicit Rule to Consult Capabilities/Context Before Architectural Assumptions

The instructions must include an explicit agent law similar in spirit to:

Before making platform-structure, endpoint, run-type, or supported-surface assumptions, consult the live boundary beginning with GET /mcp/v1/capabilities, then retrieve governed context as needed.

4) Explicit Warning Against Guessing Run Types

The instructions must clearly warn that:

run types are exact

pluralization or convention guessing is error-prone and non-canonical

unknown run types should be handled by discovery, not improvisation

The instructions must include current connection guidance for:

transport: REST/HTTP

auth: OIDC/PKCE

realm: keyhole-mcp

discovery endpoint: GET /mcp/v1/capabilities

identity endpoint: GET /mcp/v1/whoami

run dispatch surface: POST /mcp/v1/runs/start

It must also explain the difference between:

unauthenticated public discovery

authenticated identity/governed usage

read-only exploration

later proof-bearing participant flows

Functional Requirements

Agent instructions must explicitly reference GET /mcp/v1/capabilities as the initial source of truth for platform discovery.

FR-2 - Context-Before-Assumption Rule

The instruction set must require agents to retrieve governed context before making architectural or workflow assumptions where capabilities alone are insufficient.

FR-3 - Exact Run-Type Discipline

The instruction set must require exact canonical run-type usage and explicitly warn against guessing, pluralization, or convention-based inference.

FR-4 - Transport/Auth Accuracy

The instruction set must reflect the current transport/auth posture:

REST/HTTP

OIDC/PKCE

keyhole-mcp

FR-5 - Boundary-Safe Truth Model

The instructions must forbid stale-source or private-source dependency for platform truth.

FR-6 - Public vs Authenticated Surface Distinction

The instructions must clearly distinguish public discovery surfaces from authenticated governed surfaces.

FR-7 - Tombstoned Transport Awareness

The instructions must make clear that legacy SSE and JSON-RPC are not valid connection paths.

FR-8 - Agent-Friendly Error Posture

The instructions must teach agents how to behave when they encounter unknown run types, auth ambiguity, or schema uncertainty:

re-discover

re-check capabilities

consult context/schema

do not improvise hidden surface names

Implementation Guidance
A) Rewrite, Do Not Patch Around the Edges

The current file is formative and legacy. Treat this as a rehydration, not a tiny edit.

The file should be restructured around the live boundary.

B) Put the Discovery Rule Near the Top

The most important behavior change is:

start with capabilities

continue with context as needed

That rule should be visible early, not buried.

C) Make the Instruction Layer Actionable

The file should not read like abstract architecture prose.

It should tell agents what to do, in order, when they need truth.

D) Include Specific Anti-Patterns

The instructions should explicitly call out bad behavior, such as:

guessing run types

pluralizing guessed workflow names

assuming internal platform code is authoritative for external participant usage

using tombstoned transports

treating docs older than live capabilities as canonical

E) Keep It Public-Safe

The instruction layer must help agents operate correctly without exposing protected internal locks or enforcement topology.

Publish the laws, not the locks.

Risks
Risk 1 - Partial Rewrite Preserves Old Behavior

Small edits may leave legacy assumptions intact.

This story should be treated as a rehydration, not a cosmetic touch-up.

Risk 2 - Over-Specifying Unstable Details

The instructions should reflect current live posture without hardcoding unstable future assumptions that belong in later S42 stories.

Risk 3 - Teaching Too Much Internalism

The file must help agents connect and behave correctly without encouraging private-source coupling or internal lock exposure.

Risk 4 - Silent Drift Later

If the instruction layer is not clearly tied to live discovery, it will drift again.

This is why capabilities-first must be explicit.
