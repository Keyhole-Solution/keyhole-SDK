CE-V5-S42-07 — Read-Only Smoke Path
Goal

Create the first clean end-to-end read-only participant path:

discover → authenticate if needed → inspect identity → retrieve context → perform safe read-only run.

Why It Matters

The first external participant demo must succeed before proof or promotion behavior is layered on top.

Before Keyhole asks anyone to trust:

proof submission

participant contracts

governed mutation

recursive promotion

external repo participation in canonical change

it must first prove something simpler and more foundational:

an external participant can connect to the live boundary correctly and walk a truthful, read-only path end to end.

That path must be:

real

deterministic

live-boundary-based

boundary-safe

simple enough for a new developer to execute

structured enough for an agent to follow

If the first external participant cannot reliably complete the read-only path, then every later claim about recursive governance across repo boundaries becomes weaker than it should be.

This story creates the first successful governed entrance path for the developer kit.

Story Thesis

Before an external participant can safely submit proof, register contracts, or participate in a recursive governance demonstration, it must first demonstrate that it can:

discover the live MCP boundary

authenticate correctly when required

inspect identity truthfully

retrieve governed context through the published read-only surfaces

perform at least one safe read-only run against a live surface

do all of that without relying on private platform source intimacy, mock assumptions, or hidden manual intervention

That sequence is the first real external participant smoke test.

It is not a demo shortcut.

It is the first trustworthy proof that the developer kit behaves like a governed participant rather than a passive code package.

Constitutional Alignment

This story extends existing doctrine without changing platform law.

It must remain aligned with:

Rule 00 — Builders Out, Declarations In

MCP Server Boundary

Snapshot Workspace Safety

Runtime Authority Separation

Boundary Stabilization Attestation

Boundary Replay Determinism

Public Anti-Drift / selective disclosure doctrine

Truth-Source-Exclusive doctrine

This story is about safe read-only external participant verification, not about new authority or mutation.

Problem Statement

The developer kit now has, or is gaining:

boundary constitution

agent instruction rehydration

capabilities discovery

auth and identity bootstrap guidance

governed context retrieval

run-type safety and schema-discovery posture

But without a single deterministic smoke path, these remain disconnected components.

A new developer still needs an answer to the practical question:

“What is the first real thing I do to prove I am connected correctly?”

Likewise, an agent needs a concrete safe flow that says:

what to do first

what to do next

what success looks like

what common failures mean

how to recover without escalating into mutation-bearing behavior

Without this story:

onboarding stays conceptual

troubleshooting stays ad hoc

every builder invents their own first-run path

the first external participant demo is harder than it should be

later proof-bearing flows start on a shaky base

This story fixes that by creating the first clean end-to-end read-only participant path.

Story Scope

This story covers:

a smoke command or script path in the developer kit

a deterministic read-only example flow

example output capture

troubleshooting notes for common misconfiguration

documentation for the first safe external participant path

use of live MCP surfaces rather than mocks

a boundary-safe onboarding flow for both humans and agents

This story does not yet cover:

proof submission

contract registration

write-bearing runs

promotion

recursive demo orchestration across repo boundaries

mutation or runtime bridge changes

This is a read-only smoke verification story.

Non-Goals

This story must not:

simulate or mock the platform as the primary path

require platform-source browsing as a prerequisite

bypass auth posture where auth is required

perform write-bearing or proof-bearing actions

blur public discovery with authenticated governed usage

silently auto-heal major misconfigurations

expose protected internal locks or internal-only operator paths

This story must remain:

live

read-only

boundary-safe

externally reproducible

Current Boundary Truth This Story Must Reflect

The smoke path must reflect the currently disclosed and implemented participant posture.

A) Live Discovery Surface

The smoke path begins with:

GET /mcp/v1/capabilities

This is public-safe, read-only, and the first discovery surface.

B) Auth Posture

The current contract indicates:

transport: rest-http

auth flow: OIDC/PKCE

auth realm: keyhole-mcp

The smoke path must respect that posture and only authenticate when required by the subsequent step.

C) Identity Inspection Surface

The first authenticated introspection step is:

GET /mcp/v1/whoami

The smoke path must use this as the identity truth check.

D) Current Context-Access Surfaces

The smoke path should rely on currently implemented read-only context surfaces such as:

context.compile

gaps.list

lineage.get.v0_1

convergence.status.v0_1

E) Read-Only Run Safety

The smoke path must use safe read-only surfaces and must not cross into mutation-bearing behavior.

F) Live Boundary, Not Mocked Assumptions

This smoke path is only meaningful if it uses live MCP surfaces against the actual boundary.

The smoke path must implement or orchestrate:

safe read-only run execution

That sequence must be deterministic and reproducible:

retrieve capabilities

retrieve governed context

perform a safe read-only run

inspect the result

This flow must be understandable to both humans and agents.

3) Example Output Capture

read-only run result

4) Troubleshooting Notes for Common Misconfiguration

Add a practical troubleshooting section covering common early failures such as:

This should tell a user or agent how to recover without escalating into mutation-bearing experimentation.

Functional Requirements
FR-1 — End-to-End Read-Only Path

The developer kit must provide a clean end-to-end read-only path from discovery through at least one successful safe read-only run.

FR-2 — Live Surface Usage

The smoke path must use live MCP surfaces and must not rely on mocked assumptions as the canonical path.

FR-3 — Discovery-First Posture

The smoke path must begin with capabilities discovery so the participant learns the boundary before acting.

FR-4 — Auth-If-Needed Posture

The smoke path must authenticate only where required by the next step and must reflect the current OIDC/PKCE posture.

FR-5 — Identity Inspection

The smoke path must include GET /mcp/v1/whoami as the first authenticated identity verification step where authentication is required.

FR-6 — Governed Context Retrieval

The smoke path must include governed context retrieval through currently implemented read-only context surfaces.

FR-7 — Safe Read-Only Run

The smoke path must culminate in a read-only run that is safe for onboarding and does not perform mutation.

FR-8 — Troubleshooting Guidance

The smoke path must be accompanied by troubleshooting notes covering common connection, auth, and run-type issues.

FR-9 — Human and Agent Readability

The flow must be understandable and runnable by both a new human developer and an agent following structured instructions.

FR-10 — No Private-Source Dependency

The smoke path must not require reading private platform internals to understand what to do.

Required Smoke Flow

The developer kit should teach and implement a sequence equivalent to the following.

Step 1 — Discover the Boundary

Call:

GET /mcp/v1/capabilities

Purpose:

verify boundary reachability

learn the current contract

confirm transport/auth posture

inspect participant-relevant published guidance

confirm currently implemented context surfaces

This is the first truth check.

Step 2 — Authenticate If Needed

Using the published OIDC/PKCE posture for:

realm: keyhole-mcp

acquire the needed auth state for the next authenticated step.

This step should be explicit and boundary-safe.

Step 3 — Inspect Identity

Call:

GET /mcp/v1/whoami

Purpose:

confirm the current boundary identity

validate that authenticated posture is working

verify that the participant is no longer operating as an unknown unauthenticated caller

establish confidence before deeper interaction

Step 4 — Retrieve Governed Context

Invoke a current read-only context surface such as:

context.compile

or another currently published safe context-access run type.

Purpose:

retrieve current governed platform truth before action

confirm context access is operational

verify that the participant can consume the platform through the boundary

Step 5 — Perform a Safe Read-Only Run

Perform one safe read-only run using currently published and supported behavior.

A suitable example may be a context-access or status-oriented run, such as:

gaps.list

convergence.status.v0_1

another safe read-only surface disclosed by the live boundary

This step proves the participant can go beyond discovery and identity and actually use the governed boundary correctly.

Step 6 — Inspect Successful Result

Confirm that the result:

came from a live MCP surface

matches the current boundary contract

stayed read-only

did not require source intimacy or mock assumptions

Implementation Guidance
A) Build on Earlier S42 Stories

This story should consume and compose earlier work rather than bypass it:

S42-03 capabilities discovery client

S42-04 auth and identity bootstrap guidance

S42-05 context retrieval bootstrap

S42-06 run-type safety and schema-discovery posture

The smoke path is where those pieces become a coherent participant entrance flow.

B) Keep the First Path Read-Only

Do not let the first smoke path bleed into proof submission, write-bearing execution, or mutable flows.

That comes later.

The first goal is trustworthy connection and read-only success.

C) Prefer One Canonical Happy Path

The first version should not present many competing smoke flows.

Provide one canonical deterministic path first.

D) Fail Clearly

When the smoke path fails, the user or agent should know whether the issue is:

connectivity

auth

identity

context retrieval

invalid run type

malformed request shape

Avoid ambiguous “something went wrong” failure modes.

E) Show Success Explicitly

A new developer should be able to say:

I discovered the live boundary

I authenticated correctly

I confirmed who I am

I retrieved governed context

I successfully executed a safe read-only run

That is the success story this flow must enable.

Risks
Risk 1 — Too Much Scope in the First Smoke Path

If the smoke path tries to prove too much, it becomes fragile.

Keep it read-only and focused.

Risk 2 — Hidden Manual Steps

If success depends on undocumented operator interventions, the smoke path is not a real onboarding artifact.

Risk 3 — Mock-Like Behavior

If the smoke path quietly relies on local assumptions rather than live boundary calls, it weakens the entire external participant claim.

Risk 4 — Weak Troubleshooting

If failures do not map to clear causes, new developers will still struggle and the smoke path will not serve its purpose.

Risk 5 — Premature Mutation

If the flow crosses into write-bearing actions too early, it stops being a safe onboarding surface.
