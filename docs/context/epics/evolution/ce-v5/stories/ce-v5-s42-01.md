CE-V5-S42-01 - External Participant Boundary Constitution
Goal

Lock the repo split as canonical and define the external participant posture for keyhole-developer-kit.

Why It Matters

The platform must govern adjacent repos by contract and boundary, not by nesting them inside itself.

If the developer kit is treated as an internal subtree, the platform appears to work only because it has private source intimacy. That weakens the central claim.

This story exists to make the opposite true and visible:

the platform remains the governor

the developer kit remains a separate participant

platform truth is retrieved through MCP discovery and governed context

external usage does not depend on reading private platform internals

This is the first constitutional step that turns the developer kit from "a helper repo near the platform" into "the first governed external participant."

Story Thesis

keyhole-developer-kit must be treated as a distinct governed participant repository with its own lifecycle, docs, agent guidance, and onboarding posture.

Its relationship to the platform is:

not nested

not source-intimate

not implicitly privileged

not allowed to derive truth from private platform implementation details

Its canonical relationship is:

adjacent repo

governed via boundary

informed by MCP discovery/capabilities

expanded later through contract, proof, and promotion workflows

Constitutional Alignment

This story extends and operationalizes existing doctrine. It does not change platform law.

It must remain aligned with:

Rule 00 - Builders Out, Declarations In

MCP Server Boundary

Single Spine / Dual Lens / One Mint

Snapshot Workspace Safety

Capability Preservation

Runtime Authority Separation

Public Anti-Drift / selective disclosure doctrine

Boundary Stabilization Attestation

MCP Boundary Closure

This story is specifically about boundary posture, not feature addition.

Problem Statement

Historically, the SDK/developer-kit lived nested inside the platform repository. That posture created architectural ambiguity:

the governed participant and the governor appeared physically merged

instructions and assumptions could silently depend on private platform source

agent guidance could drift toward reading internals instead of retrieving governed context

the repo boundary was unclear for future builders and demo participants

Now that keyhole-developer-kit has been cut out into its own private repository, the architecture must formally recognize that split and remove any remaining legacy assumptions.

Without this story, the split is merely physical.

With this story, the split becomes constitutional and operational.

Story Scope

This story covers:

explicit documentation that keyhole-developer-kit is a separate governed participant

formal ownership and relationship statement between the platform repo and the developer-kit repo

removal of any remaining nested-repo assumptions in docs, guidance, and instructions

establishment of MCP discovery/capabilities as the first truth surface for the developer kit

canonical repo-boundary language in developer-kit documentation

This story does not yet implement:

full MCP client behavior

auth bootstrap

context retrieval helpers

contract registration

proof bundle submission

recursive demo orchestration

Those follow in later S42 stories.

Non-Goals

This story must not:

redesign the platform boundary

expand the MCP surface

introduce new run types

define proof submission semantics

grant the developer kit internal platform authority

re-couple the developer kit to platform source trees

imply that the developer kit can learn the platform by reading private platform code

create hidden privileged paths for Lance or any future external participant

This is a boundary constitution story, not a runtime-integration story.

Current Truth This Story Must Lock In

The following must be treated as canonical inputs:

keyhole-developer-kit now lives in its own separate private repository

the platform's public discovery surface is GET /mcp/v1/capabilities

capabilities disclosure is the first public-safe truth surface for external participants

the MCP boundary is already attested closed

the boundary is already stabilized and replay-deterministic

legacy SSE and JSON-RPC transports are tombstoned

the canonical boundary posture is REST/HTTP plus OIDC/PKCE, with boundary rules and implemented surfaces disclosed through capabilities

This story must make those truths legible in repo posture and documentation.

Deliverables
1) Explicit Repo-Boundary Doctrine for keyhole-developer-kit

Create or update a canonical boundary note that states:

keyhole-developer-kit is a separate governed participant repo

it is not part of the platform source tree

it must learn the platform through MCP discovery and governed context

private platform source intimacy is non-canonical for developer-kit operation

This doctrine should be visible in repo documentation, not hidden in a private planning note.

2) Ownership and Relationship Statement

Add a clear relationship statement describing:

what the platform repo is

what the developer-kit repo is

how they relate

what each is allowed to assume about the other

how governance crosses the boundary

This statement must make clear:

the platform governs

the developer kit participates

boundary truth is surfaced through MCP, not private implementation reach-through

3) Removal of Nested-Repo Assumptions

Find and remove or rewrite any remaining guidance that implies:

the SDK/developer kit lives inside keyhole_Platform

the developer kit can depend on local relative paths into the platform repo

platform source files are the canonical place to learn external-usage truth

developer-kit contributors should inspect private internals first

This includes repo docs, setup notes, agent instructions, and any developer onboarding text.

4) Canonical Boundary Note in Repo Docs

Add a short, high-visibility note in developer-kit docs that says the first truth surface is:

GET /mcp/v1/capabilities

and that all external participant understanding should begin there before assumptions are made.

Functional Requirements
FR-1 - Separate Participant Posture

The developer kit must be explicitly described as a separate governed participant repo rather than a subcomponent of platform source.

FR-2 - Boundary-First Truth

The developer kit must instruct humans and agents to consult MCP discovery/capabilities before making assumptions about platform structure, interfaces, or supported behavior.

FR-3 - No Source Intimacy Requirement

The repo must declare private platform source intimacy non-canonical for normal developer-kit usage, onboarding, and agent behavior.

FR-4 - Relationship Clarity

The repo must include a clear explanation of the boundary relationship between:

keyhole_Platform

keyhole-developer-kit

including what is shared conceptually and what is intentionally separated operationally.

FR-5 - Legacy Assumption Removal

No instructions may remain that imply the developer kit is nested inside platform source or should use nested relative paths or internal platform browsing as its primary discovery method.

FR-6 - Public-Safe Posture

The developer-kit boundary documentation must stay public-safe in doctrine terms:

laws and outcomes may be described

protected enforcement internals must not be exposed as onboarding requirements

Implementation Guidance
A) Establish a Canonical Boundary Statement

Add a concise statement similar in spirit to:

keyhole-developer-kit is the first governed external participant repository. It is separate from keyhole_Platform and must learn platform truth through the MCP boundary, beginning with capabilities discovery, rather than through private platform source intimacy.

This should appear in a stable, visible repo location.

B) Add a Relationship Section

Document:

the platform as governor

the developer kit as participant

later contract/proof participation as future-facing

current discovery posture as capabilities-first

This should reduce ambiguity for both humans and agents.

C) Remove Local-Tree Bias

Audit for language that assumes:

co-location in the same filesystem tree

relative imports or paths into platform source

reading platform source as a prerequisite for usage

nested dev workflows as canonical

Replace such language with boundary-safe language.

D) Point to Capabilities as First Truth Surface

Docs and agent guidance should explicitly say:

start with GET /mcp/v1/capabilities

use that surface to discover transport, auth posture, available read-only surfaces, and client guidance

do not guess hidden or future surfaces

E) Prepare the Repo for Later Stories

This story should make later work easier by establishing the repo posture needed for:

capabilities client

auth bootstrap

context retrieval

run-type safety

smoke paths

recursive demo participation

But it must avoid prematurely hardcoding unstable integration details.

Risks
Risk 1 - Physical Split Without Mental Split

The repo may be separated physically while docs and agent guidance still behave as if it is nested.

This story exists to eliminate that risk.

Risk 2 - Over-Specifying Later Integration

It would be easy to prematurely define contract/proof details that belong to later S42 stories.

This story must stop at posture and boundary law.

Risk 3 - Hidden Legacy Guidance

Old setup notes or agent instructions may still quietly reference nested assumptions.

The audit step is therefore essential.
