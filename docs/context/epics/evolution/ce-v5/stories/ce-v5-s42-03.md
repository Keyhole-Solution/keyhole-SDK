CE-V5-S42-03 — Capabilities Discovery Client
Goal

Implement a minimal client path in the developer kit that retrieves and normalizes the public-safe boundary contract from the capabilities surface.

Why It Matters

The developer kit must be able to learn the boundary from the boundary itself.

If the first external participant cannot retrieve and normalize the live capabilities contract, then all later behavior remains fragile:

onboarding depends on stale docs

agents guess supported surfaces

transport/auth posture drifts

context-access assumptions become folklore

compatibility decisions become manual

launch demos depend on human memory instead of governed discovery

That is unacceptable for the first governed external participant.

This story creates the first concrete machine path through which keyhole-developer-kit learns:

what contract it is speaking

what transport and auth posture apply

what compatibility floor applies

what context-access surfaces are currently implemented

what public-safe guidance the boundary itself is publishing

This is the first executable proof that the external participant can discover the platform from the platform itself.

Story Thesis

The first external participant must not begin from repo lore.

It must begin from live boundary disclosure.

That means keyhole-developer-kit must be able to:

call GET /mcp/v1/capabilities

validate that the response is a usable mcp/v1 capabilities document

normalize it into a stable internal model

preserve discovery metadata and digest-like fields where available

expose a clean API for downstream consumers without requiring them to parse raw response shape

cache discovery snapshots safely enough for local developer ergonomics without teaching stale truth

This story is the first executable layer after the instruction-layer rehydration.

It turns:

“capabilities-first discovery” as doctrine

into:

a real client primitive inside the developer kit

Constitutional Alignment

This story extends existing doctrine without changing platform law.

It must remain aligned with:

Rule 00 — Builders Out, Declarations In

MCP Server Boundary

Single Spine / Dual Lens / One Mint

Snapshot Workspace Safety

Runtime Authority Separation

Capability Preservation

Boundary Replay Determinism

Boundary Stabilization Attestation

Public Anti-Drift / selective disclosure doctrine

This story is about boundary consumption, not boundary redesign.

Problem Statement

The current capabilities surface already discloses a rich public-safe contract.

But without a discovery client in the developer kit, the external repo still lacks a first-class machine entrypoint for learning that contract.

Without this story:

every builder must hand-inspect raw JSON

every agent must re-derive meanings from ad hoc parsing

downstream helpers will each invent their own parsing logic

caching behavior will be inconsistent

boundary metadata may be lost

compatibility posture will not have a canonical normalized representation

That would recreate the exact drift this epic is trying to eliminate.

The developer kit needs one small, clear, canonical discovery client that all later helpers can build on.

Story Scope

This story covers:

a capabilities fetch client inside keyhole-developer-kit

a normalized capabilities model

caching and digest-aware handling for discovery snapshots

helpers for extracting compatibility, context-access, and client-guidance data

preservation of generated-at / digest / correlation / context-reference metadata where present

a developer-safe API surface for later S42 stories

This story does not yet cover:

auth token acquisition

identity bootstrap

context retrieval helper flows

run dispatch

schema discovery implementation

proof submission

contract registration

recursive demo orchestration

Those are downstream stories.

Non-Goals

This story must not:

redesign GET /mcp/v1/capabilities

hardcode private platform implementation details

invent capabilities not present in the boundary contract

treat cached discovery as more authoritative than live discovery

expose hidden or protected enforcement topology

mutate runtime state

implement full SDK generation from the capabilities contract

require platform-source reading to interpret discovery

This is a minimal discovery client, not a generator or orchestration engine.

The client must reflect the current public-safe capabilities posture already disclosed by the boundary.

A) Canonical Discovery Endpoint

The initial source of truth is:

GET /mcp/v1/capabilities

D) Current Public-Safe Client Guidance

The capabilities contract also includes public-safe guidance for:

Deliverables
1) Capabilities Fetch Client

Implement a minimal fetch client in the developer kit that calls:

GET /mcp/v1/capabilities

This client must:

perform read-only discovery

parse the response safely

return a normalized structure rather than requiring raw JSON consumers everywhere

fail clearly when the response shape is unusable

2) Normalized Capabilities Model

Implement a stable internal representation of the capabilities result.

The normalized model must expose, at minimum:

contract identity

transport/auth posture

compatibility posture

feature flags

schema versions

implemented context-access surfaces

client guidance

discovery metadata

This model should be convenient for later stories to consume without forcing them to understand the full raw response structure.

3) Caching / Digest Handling for Discovery Snapshots

Implement lightweight discovery snapshot handling so local consumers can:

cache the most recent successful capabilities result

preserve relevant digest or timestamp data

know when a cached result exists

avoid teaching that cached output is more authoritative than live discovery

Caching is for ergonomics and repeatability, not for overriding boundary truth.

4) Helper for Compatibility, Context-Access, and Client Guidance Extraction

Provide helper methods that expose the most important participant-facing slices of the normalized model, including:

compatibility data

context-access surfaces

auth/transport posture

run-type guidance

gap-workflow guidance

event-query guidance

This gives later S42 stories a clean base primitive.

FR-1 — Live Capabilities Retrieval

FR-3 — Metadata Preservation

FR-4 — Helper Extraction

Downstream consumers must be able to retrieve:

compatibility posture

context-access surfaces

client-guidance data

from helper methods over the normalized model.

FR-5 — Public-Safe Consumption

The client must operate entirely on the public-safe capabilities contract and must not depend on platform-source intimacy.

FR-6 — Cache Without Truth Drift

Caching must be supported for local convenience, but the design must keep live discovery authoritative.

FR-7 — Boundary-Safe Failure Mode

If the capabilities response is malformed, unavailable, or incomplete, the client must fail clearly and conservatively rather than inventing missing surfaces.

FR-8 — Minimal Dependency Footprint

The first version should remain minimal and not introduce unnecessary complexity or a large dependency stack.

Implementation Guidance
A) Keep the First Client Small and Exact

This story should implement the smallest useful discovery client that works reliably.

Do not expand into full orchestration or SDK generation.

B) Normalize Around Participant Needs

The normalized model should be shaped around what downstream developer-kit consumers need most:

“How do I connect?”

“What contract am I talking to?”

“What context surfaces exist?”

“What guidance did the boundary publish?”

C) Preserve Metadata, Do Not Discard It

The raw response includes useful meta/discovery fields.

Do not strip them away just because the first consumer does not yet use all of them.

They are important for:

traceability

cache inspection

future debugging

discovery snapshot comparison

D) Cache Conservatively

Caching should be local and explicit.

It should never silently replace live truth as canonical.

A cached snapshot is a convenience artifact, not a governing truth source.

E) Fail Closed Under Uncertainty

If parsing fails or the response shape is incomplete, the client should fail clearly.

Do not fill in guessed defaults for:

run types

supported surfaces

auth posture

feature support

Discovery must prefer truthful incompleteness over fabricated completeness.

F) Make Later Stories Easy

This story should set up later work by making it easy for future helpers to ask questions like:

what is the current minimum SDK version?

what context surfaces are implemented?

does the boundary require charter?

what auth flow is expected?

what is the published run-type guidance?

Cache / Snapshot Contract

The caching posture should obey these rules.

Rule 1 — Live Discovery Is Authoritative

The live call to GET /mcp/v1/capabilities remains authoritative.

Rule 2 — Cache Is Advisory

Cached snapshots are local developer aids, not truth overrides.

Rule 3 — Preserve Timestamps / Digests

Where discovery metadata is present, the cache must preserve it.

Rule 4 — No Silent Fabrication

A cache miss or stale cache must not cause the client to invent missing capabilities.

Rule 5 — Local File Only

The initial cache should remain local to the developer-kit environment and should not introduce unnecessary shared-state complexity.

Risks
Risk 1 — Overbuilding the First Client

It would be easy to turn this into a general SDK generator or orchestration layer.

This story must remain minimal and discovery-focused.

Risk 2 — Throwing Away Important Metadata

If normalization is too aggressive, later traceability and debugging become harder.

Preserve metadata early.

Risk 3 — Teaching Cache as Truth

If the cache behavior is unclear, builders may treat old discovery as canonical.

The design must keep live discovery authoritative.

Risk 4 — Coupling to Current Response Shape Too Rigidly

The normalized model should be stable for consumers while still tolerating the live response evolving within contract bounds.

Risk 5 — Guessing Missing Fields

The client must not compensate for malformed discovery by inventing missing support.

Truthful incompleteness is safer than fabricated completeness.
