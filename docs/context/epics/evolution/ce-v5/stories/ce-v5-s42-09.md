CE-V5-S42-09 - Recursive Demo Readiness Pack
Goal

Prepare the developer kit to serve as the first external participant in the recursive governance demonstration.

Why It Matters

This is the bridge between developer onboarding and the DEV-UX recursive proof story.

Up through the earlier S42 stories, the developer kit becomes increasingly real as an external participant:

it is split cleanly from the platform repo

it learns the boundary through capabilities

it bootstraps auth and identity correctly

it retrieves governed context before acting

it validates run types and request shapes

it can complete a safe read-only smoke path

it contains proof-ready scaffolding for later contract/proof participation

That is necessary groundwork.

But none of it becomes strategically persuasive until the developer kit can participate in the actual recursive governance demonstration.

This story prepares that bridge.

It does not yet run the whole platform-side recursive demonstration.

It prepares the external side so that when DEV-UX proof, contract, verdict, and console surfaces are ready, the developer kit can step into the flow cleanly, deterministically, and without improvisation.

Without this story, the external participant side remains a set of good ingredients.

With this story, it becomes a ready-to-run half of the first governed cross-boundary proof.

Story Thesis

The first recursive governance demonstration must not rely on improvised external behavior.

The external participant side must already know:

which demo-safe change path to use

how to retrieve boundary truth before acting

how to prepare a candidate change in a governed repo

how to run the external-side verification steps

where the handoff to platform-side proof and verdict flow begins

what platform-side evidence should appear if the demo is working correctly

That means the developer kit needs a recursive demo readiness pack:

a candidate demo workflow

a small safe feature/change path

operator notes

explicit mapping from developer-kit actions to expected platform evidence

This story creates that pack.

It makes the external participant side demoable on purpose rather than "probably demoable if the operator remembers everything."

Constitutional Alignment

This story extends existing doctrine without changing platform law.

It must remain aligned with:

Rule 00 - Builders Out, Declarations In

MCP Server Boundary

Capability Preservation

Boundary Replay Determinism

This story is about demo readiness and choreography, not new mutation authority.

Problem Statement

The recursive governance claim is strongest when it is shown across a real repo boundary:

story exists

external repo retrieves governed truth

external repo changes something safely

verification artifacts are produced

proof-bearing handoff occurs

the platform evaluates, returns verdicts, and promotes or rejects accordingly

evidence appears in the expected places

humans can see the whole loop

The platform-side DEV-UX stories are building the governance engine for that.

But the developer-kit side also needs a prepared and disciplined demo posture.

Without this story, the demo risks several failure modes:

Failure Mode 1 - External Demo Improvisation

The operator invents the participant-side workflow during the demo.

That makes the flow:

non-deterministic

fragile

hard to repeat

hard to document

hard to trust

Failure Mode 2 - Oversized Demo Change

The first external-side change touches too much surface area and turns the demo into a debugging session instead of a proof.

Failure Mode 3 - Unclear Handoff

The developer-kit side finishes its part, but the handoff into DEV-UX proof/promotion flow is ambiguous.

That weakens the recursive governance story.

Failure Mode 4 - Missing Evidence Expectations

The operator does not know what platform-side evidence should appear after each participant action.

That makes the demo harder to validate and explain.

This story fixes those issues by packaging the external participant side into a deterministic readiness pack.

Story Scope

This story covers:

a candidate recursive demo workflow in the developer kit

a small demo-safe feature/change path

operator notes for the external-side half of the recursive demonstration

explicit mapping from developer-kit actions to expected platform-side evidence

documentation of handoff points from external participant actions into DEV-UX proof/promotion flows

preparation of the developer kit to be the first external participant in the recursive governance story

This story does not yet cover:

final platform-side proof submission implementation

final contract registration implementation

final verdict/promotion execution

production launch sealing

marketing/video recording execution itself

Those belong to DEV-UX completion and later launch work.

Non-Goals

This story must not:

overbuild the demo into a large product scenario

choose a change path that requires fragile manual intervention

define platform proof or promotion law from the participant side

hardcode private platform internals into demo logic

blur the line between "demo-ready participant" and "platform-side recursive closure complete"

assume unfinished DEV-UX surfaces are already sealed

This is a readiness pack story, not the final demo execution story.

Current Truth This Story Must Respect

This story must remain grounded in what the developer kit can already prepare and what DEV-UX is expected to provide.

A) The Developer Kit Is the First External Participant

The repo is the correct candidate for the first cross-boundary demonstration because it is:

separate from the platform repo

boundary-consuming

designed to learn through MCP

intended to become the first governed external participant

B) Earlier S42 Work Already Establishes the Entrance Posture

The developer kit can already be prepared to:

discover capabilities

bootstrap auth and identity

retrieve governed context

validate run-type usage

complete a read-only smoke path

hold proof-ready scaffolding

This story composes those pieces.

C) DEV-UX Owns the Platform-Side Recursive Closure

The platform-side surfaces expected to receive the external handoff include:

participant contract registry

proof submission pipeline

verification graph / blast-radius resolution

structured verdict and repair artifacts

governance console

recursive demo flow

This story must prepare the developer kit to hand off cleanly into those flows without redefining them.

D) Demo Success Must Be Small, Real, and Repeatable

The first external-side demo should use:

a small safe change

a deterministic workflow

visible evidence expectations

explicit handoff points

The goal is operational undeniability, not demo complexity.

Deliverables
1) Candidate Demo Workflow in the Developer Kit

Define a canonical external-side workflow for the recursive governance demo.

This workflow must describe, in order:

retrieve capabilities

retrieve governed context

confirm participant posture

apply a small demo-safe change in the developer kit

run local verification or verification-runner scaffold

assemble the external-side proof-ready artifacts or placeholders

hand off to platform-side contract/proof surfaces when available

observe expected verdict/promotion/evidence behavior on the platform side

The flow must be explicit and deterministic.

2) Small Demo-Safe Feature/Change Path

Choose and document a minimal demo-safe change path inside the developer kit.

This change path must be:

small

easy to verify

low blast radius

meaningful enough to prove governance across boundaries

not so trivial that it looks fake

not so large that it becomes risky or noisy

Good examples conceptually include:

a small new command or subcommand

a tiny visible behavior addition

a bounded documentation-plus-code surface where verification is clear

a minimal public-safe enhancement with obvious expected outputs

The exact demo change should be selected to maximize clarity and minimize demo fragility.

3) Operator Notes for Running the External-Side Half of the Recursive Demo

Provide a runbook or operator note set for the external participant side.

These notes must tell the operator:

where to start

what commands or steps to run

what preconditions to check

what evidence or output should appear after each stage

where the participant-side flow stops and platform-side DEV-UX flow begins

how to identify common failure modes during the demo

This runbook must be usable by a human operator without relying on unstated tribal memory.

4) Mapping from Developer-Kit Actions to Expected Platform-Side Evidence

Provide a mapping from each major developer-kit-side action to the expected platform-side observation.

For example, conceptually:

capabilities retrieval -> visible discovery success / context posture

context retrieval -> context snapshot or read-only evidence expectations

candidate change preparation -> participant-side provenance expectation

verification run -> proof-ready local output expectation

future submission handoff -> expected platform-side contract/proof intake evidence

platform evaluation -> expected verdict artifact

successful promotion path -> expected evidence or console visibility

rejection path -> expected remediation artifact / gap linkage

This mapping is critical because it lets the demo be validated, not merely narrated.

5) Explicit Handoff Definition to DEV-UX Flows

Define the exact handoff points where external participant work transitions into platform-side recursive-governance logic.

At minimum, make explicit:

where contract registration would occur

where proof submission would occur

where verification graph resolution begins

where verdict artifacts are expected

where console visibility should be checked

This prevents the demo from collapsing into "and then the platform somehow does the rest."

Functional Requirements
FR-1 - Scriptable External Demo Flow

The developer kit must have a candidate external-side recursive demo workflow that can be followed deterministically.

FR-2 - Demo-Safe Change Path

A small, bounded, repeatable feature/change path must be identified and documented for the external-side demonstration.

FR-3 - Operator Runbook

The repo must include operator notes for executing the external participant half of the recursive demo.

FR-4 - Evidence Mapping

The developer kit must document the mapping between participant-side actions and expected platform-side evidence.

FR-5 - Explicit Handoff Points

The handoff points from the developer kit into DEV-UX contract/proof/promotion flows must be clearly identified.

FR-6 - Supported vs Future Distinction

The readiness pack must distinguish between:

what the developer kit can execute now

what will be completed when DEV-UX proof/promotion surfaces are ready

FR-7 - Human and Agent Readability

The workflow and notes must be clear enough for both a human operator and an agent-driven runbook consumer.

FR-8 - No Private-Source Dependency

The workflow must not depend on reading private platform internals to know how the external participant should behave.

Implementation Guidance
A) Keep the First Demo Small

The first recursive governance demonstration must prove the loop, not overwhelm the viewer.

Choose a demo change that is:

easy to understand

easy to verify

visibly real

low blast radius

B) Prefer Determinism Over Breadth

The external-side workflow should have one canonical happy path first.

Multiple demo branches can come later.

C) Make the Handoff Explicit

One of the most important parts of this story is to prevent a fuzzy handoff into platform-side DEV-UX flows.

The external-side readiness pack must explicitly say:

"this is where the participant stops"

"this is where the platform-side proof flow begins"

D) Document Evidence Expectations Before the Demo Runs

Do not wait until after the demonstration to decide what counts as success.

The expected evidence map should be defined ahead of time.

E) Do Not Pretend Unfinished DEV-UX Pieces Are Done

If a step depends on a planned platform-side surface, label it clearly as a future handoff point.

This story is about readiness, not fiction.

F) Build on Earlier S42 Stories

This story should compose earlier work rather than bypass it:

capabilities discovery

auth/identity bootstrap

context retrieval

run-type safety

read-only smoke path

proof-ready scaffolding

The readiness pack is where those become the external-side half of the recursive story.

Candidate Demo Selection Guidance

The chosen demo-safe change should satisfy these constraints.

Constraint 1 - Real Boundary Value

The change must be meaningful enough that its governance across a repo boundary is obviously real.

Constraint 2 - Low Blast Radius

The change must not risk destabilizing the developer kit or the demo itself.

Constraint 3 - Easy Verification

There must be a clear way to show:

what changed

how it was verified

what evidence it produced

Constraint 4 - Good Handoff Shape

The change should fit naturally into future contract/proof/promotion flows.

Constraint 5 - Public-Safe Storytelling

The demo should be explainable publicly without exposing protected internal locks.

The readiness pack docs must include:

A) What the Recursive Demo Is Proving

Explain that the demo is proving:

cross-boundary governed change

visible recursive governance behavior

B) What the Developer Kit Side Owns

Explain that the external participant side owns:

discovery

C) What DEV-UX Owns

Explain that platform-side DEV-UX owns:

D) What Success Looks Like

Define what counts as success for the external side and what counts as success once the platform side picks up the handoff.

E) Common Failure Modes

Document likely issues such as:

wrong demo branch or workspace

invalid run-type or schema assumptions

local verification runner mismatch

unclear handoff to unfinished DEV-UX surfaces

missing expected platform-side evidence

Risks
Risk 1 - External Demo Overcomplexity

If the first demo-safe change is too large, the demo becomes harder to explain and easier to break.

Risk 2 - Fuzzy Handoff

If the external-side pack does not define where DEV-UX takes over, the demo story becomes muddy.

Risk 3 - Evidence Ambiguity

If expected platform-side evidence is not documented up front, the demo becomes harder to validate.

Risk 4 - Pretending Planned Surfaces Are Finished

If the readiness pack glosses over unfinished DEV-UX surfaces, it will overpromise and confuse future builders.

Risk 5 - Excessive Dependence on Human Memory

If the runbook is weak, the demo will still rely on private operator knowledge rather than becoming repeatable.
